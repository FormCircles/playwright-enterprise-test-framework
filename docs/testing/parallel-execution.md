# Parallel Test Execution

## Overview

The Playwright Enterprise Test Framework supports experimental parallel test execution using `pytest-xdist`.

Parallel execution can reduce test duration as the framework grows, but it also introduces risks when multiple test workers interact with the same backend application and shared test data.

The current API test suite uses:

* A shared FastAPI mock application
* Shared backend device storage
* Backend-generated numeric resource IDs
* Per-test entity cleanup
* An optional suite-level backend reset

Because pytest-xdist workers run independently while sharing the same backend, API integration tests are not fully isolated from one another.

## Installation

Install the framework dependencies:

```bash
pip install -r requirements.txt
```

The framework currently uses:

```text
pytest-xdist==3.8.0
```

Confirm xdist is available:

```bash
pytest --help | grep -E -- '--numprocesses|--dist'
```

## Serial Execution

Serial execution remains the default and safest execution model:

```bash
pytest -m "api and regression" \
  --env=local
```

Serial execution ensures that tests interact with the backend one at a time.

## Supported Parallel Execution Model

The currently supported experimental model for API regression tests is:

```bash
pytest -m "api and regression" \
  --env=local \
  -n 2 \
  --dist loadscope
```

This configuration uses:

* Two pytest workers
* Module-level scheduling
* Sequential test execution within each module

With the current suite structure, execution is generally distributed as follows:

```text
Worker gw0
└── api_tests/tests/test_devices.py

Worker gw1
└── api_tests/tests/test_devices_api.py
```

The stateful device CRUD tests remain together on one worker, reducing the risk that their setup and cleanup operations overlap with one another.

## Validation Results

### Serial baseline

The API regression suite was executed serially:

```text
9 tests passed
```

Observed wall-clock time:

```text
Approximately 4.22 seconds
```

### Two workers with `loadscope`

The API regression suite was executed using:

```bash
pytest -m "api and regression" \
  --env=local \
  -n 2 \
  --dist loadscope
```

Validation included ten consecutive runs:

```text
10 successful runs
90 total test executions
0 failures
```

Individual test runs completed in approximately:

```text
2.96 to 7.71 seconds
```

The parallel configuration did not provide a consistent performance improvement for the current small suite.

### Two workers with `load`

The API regression suite was also tested using the default load scheduler:

```bash
pytest -m "api and regression" \
  --env=local \
  -n 2 \
  --dist load
```

A shared-state race condition occurred during repeated execution.

Observed failure:

```text
test_get_device_by_id
Expected status: 200
Actual status: 404
Response: {"detail":"Device not found"}
```

A device created for the test existed initially but was no longer available when the test attempted to retrieve it.

This demonstrates that unrestricted per-test distribution is not safe with the current shared backend.

## Shared-State Risks

### Shared backend storage

All xdist workers connect to the same FastAPI mock application:

```text
Worker gw0 ─┐
            ├── Shared FastAPI backend
Worker gw1 ─┘
```

The workers do not receive isolated databases, storage collections, or backend instances.

Actions performed by one worker can therefore affect another worker.

### Backend-generated resource IDs

Device IDs are generated by the backend.

For example:

```text
Worker gw0 creates device ID 3.
Worker gw1 modifies, deletes, or resets shared backend data.
Worker gw0 requests device ID 3.
Backend returns 404.
```

Unique device names reduce naming collisions but do not isolate numeric IDs or shared backend records.

### Per-test cleanup

The framework tracks test-created resources and deletes them during fixture teardown.

This works reliably during serial execution, but parallel workers can perform cleanup concurrently.

Potential risks include:

* One worker deleting a resource another worker still needs
* Resource IDs being reused by the backend
* Cleanup operations running in an unexpected order
* Cleanup receiving 404 because another operation already deleted the record
* Leftover records when a worker terminates before teardown

The existing cleanup implementation accepts HTTP 404 during deletion because the desired final state is that the entity no longer exists. However, accepting 404 does not prove that another worker did not delete the resource prematurely.

### Suite-level backend reset

Pytest session-scoped fixtures execute separately in each xdist worker.

Without protection, each worker could attempt to call:

```text
POST /test/reset
```

A reset from one worker could remove backend data while another worker is still running tests.

The framework therefore skips suite-level backend reset inside xdist workers.

During parallel execution, isolation depends primarily on per-test cleanup.

The suite-level reset is not currently coordinated to execute exactly once after all workers finish.

## Unsupported Execution Models

The following configurations are not currently supported for API integration tests:

```bash
pytest -n 2 --dist load
```

```bash
pytest -n auto
```

```bash
pytest -n 4
```

The default `load` scheduler may distribute related stateful tests across multiple workers.

`-n auto` may create more workers than the backend and cleanup model can safely support.

Increasing the worker count does not create backend isolation.

## Framework Unit Tests

Framework unit tests are generally safer for parallel execution because they do not call the shared application backend.

Examples include tests for:

* Assertion helpers
* Configuration loading
* Environment guards
* Logging sanitization
* Failure diagnostics
* JSON schemas
* Test data factories
* Backend reset response evaluation

These tests mostly use local Python objects, mocks, and temporary files.

They can be evaluated separately using:

```bash
pytest -m "not api and not ui" \
  -n auto
```

This command should still be validated before becoming a CI default.

## UI Tests

Parallel execution of UI tests has not yet been approved.

UI tests introduce additional concerns:

* One browser instance per worker
* Increased CPU and memory usage
* Shared backend test data
* Screenshot and diagnostic filename collisions
* Login/session interactions
* Browser trace and video artifact handling

UI parallelism should be evaluated in a separate story.

## Current Recommendation

Use serial execution as the default for CI and normal local development:

```bash
pytest --env=local
```

Use parallel API execution only for controlled evaluation:

```bash
pytest -m "api and regression" \
  --env=local \
  -n 2 \
  --dist loadscope
```

Current rules:

1. Limit API execution to two workers.
2. Use `--dist loadscope`.
3. Do not use the default `load` scheduler.
4. Do not use `-n auto` for API integration tests.
5. Skip suite-level backend reset inside workers.
6. Retain per-test cleanup.
7. Investigate any unexpected 404 response as a possible race condition.
8. Do not enable xdist globally through `pytest.ini`.
9. Do not enable API parallelism in CI until backend isolation improves.
10. Reevaluate performance after the API suite becomes significantly larger.

## Performance Considerations

The current API regression suite contains only nine tests.

Worker startup, test collection, process communication, fixture initialization, and Playwright request-context creation introduce overhead.

Observed parallel execution was not consistently faster than serial execution.

Parallelism should not be enabled solely because it is technically available. It should be adopted when:

* The suite is sufficiently large
* Execution time improves consistently
* Shared state is isolated
* Cleanup is deterministic
* Repeated runs are stable
* CI resource consumption remains acceptable

## Future Improvements

The following approaches could make broader parallel execution safe.

### Worker-specific backend namespaces

Include the xdist worker ID in test-created resources:

```text
gw0_TEST_Device_<unique-value>
gw1_TEST_Device_<unique-value>
```

This improves ownership visibility but does not fully prevent numeric-ID collisions or global resets.

### Worker-specific databases

Assign each worker a separate database or storage namespace:

```text
gw0 → test database 0
gw1 → test database 1
```

This would provide stronger isolation.

### One backend instance per worker

Start a separate FastAPI mock application instance for each worker:

```text
gw0 → http://127.0.0.1:8081
gw1 → http://127.0.0.1:8082
```

Each worker would receive its own base URL and backend state.

### Ownership-aware cleanup

Store worker ownership alongside each created entity and verify ownership before deletion.

Cleanup should delete only entities created by the current test or worker.

### Controller-only backend reset

Coordinate the backend reset so it executes once from the pytest controller after all workers complete.

This requires an xdist-aware lifecycle implementation rather than a normal worker session fixture.

### Worker-aware diagnostics

Failure diagnostics should include the worker identifier:

```text
Worker: gw0
```

This would make parallel race conditions easier to investigate.

### Dedicated parallel test environment

A future CI environment could provide disposable, isolated backend instances specifically for parallel execution.

## Decision

Parallel API execution is conditionally supported for local evaluation with:

```text
-n 2 --dist loadscope
```

The current framework is not safe for unrestricted per-test parallel distribution.

Serial execution remains the official default until backend state, cleanup, and reset behavior are fully isolated across workers.
